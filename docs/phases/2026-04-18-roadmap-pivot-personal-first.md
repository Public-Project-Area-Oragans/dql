## 1. 변경 요점

### 1.1 기존 결정 (지금까지)

Phase 2 설계 §10은 다음 게이트를 두었다:

> 3챕터 중 1챕터는 게임 시뮬레이터 금지, 마크다운 원문으로만 일주일 학습.
> 나머지 2챕터는 게임 시뮬레이터 + 이론 탭 병행.
> 일주일 뒤 본인이 (a) 핵심 개념 3가지, (b) 다이어그램 스케치, (c) 학습 방식 자기 태그를 작성.
> 대조군(마크다운) 밀도가 게임 2챕터보다 같거나 높으면 **가치 명제 실패 신호**.

이 결정의 전제는 "1주일 슬라이스 체감 ≒ 프로젝트 지속 가치의 신호"였다.

### 1.2 변경된 결정 (2026-04-18)

사용자 지시로 다음 순차 구조로 변경:

1. **개인 사용용 전체 기능 완성 (Phase Personal-Complete, 이하 P0)**
2. **모든 언어 카테고리를 개인 학습용으로 실사용 (P1)**
3. **회사/팀 사용 확장 버전 (P2)**
4. **불특정 다수 공용 버전 (P3)**

각 단계는 **완료 후 다음 단계 계획**을 별도 설계 문서로 착수한다. 1주일 대조군 실험 게이트는 **제거**. 이유: 실측 전에 슬라이스 완성도만으로 프로젝트 방향을 끊는 것보다, 완성 이후 축적된 실제 사용 데이터로 판단하는 것이 정보 가치가 크다는 판단.

## 2. 단계별 정의

### P0 — 개인 사용용 전체 기능 완성

**목표**: 원 설계 문서(2026-04-17)의 모든 Task가 동작하고, 5개 배포 카테고리(java-spring / dart / flutter / mysql / msa)에서 결함 없이 학습 가능한 상태.

**범위 (현재까지 완료된 것)**:
- Phase 1 Task 1~11 + Task 12 (이론 뷰 ASCII/표/mermaid → 순수 Flutter 위젯)
- Phase 2 Task 2-0 ~ 2-6c + 2-1 (MSA 2챕터 구조 조립 시뮬레이터, 대조군 step1)
- Task 2-5 (계측 로그)
- Phase 3 다이어그램 위젯 이주 PR #28~#35 (이미지 미사용, 전 카테고리)

**P0 잔여 항목 (미정 — 실사용 중 발견되는 결함/요청을 적재)**:
- [ ] `integration` 잡 `continue-on-error: true` 제거 (5회 연속 녹색 누적됨)
- [ ] Phase 3 flowchart 파서 잔여 "other" 실패 19건 샘플 분석 → 필요 시 파서 보강
- [ ] Phase 2 구조 조립 시뮬레이터 `docs-source` 전체 MSA 기타 챕터로 override 확장 여부 결정 (현재 2챕터만)
- [ ] 타 카테고리(java-spring/dart/flutter/mysql) 각각 최소 1개 도메인별 시뮬레이터 파일럿:
  - java-spring: 아직 시뮬레이터 없음 → 후속 타입 설계 필요(예: Spring Bean 의존 그래프 조립)
  - dart: 시뮬레이터 없음 → 후속
  - flutter: 시뮬레이터 없음 → 후속 (Widget 트리 조립?)
  - mysql: 시뮬레이터 없음 → 후속 (SQL 실습? ER 조립?)
- [ ] 모든 카테고리에 대해 본인 학습을 실제로 일정 범위 진행해 실제 결함 목록 확정

**P0 → P1 전환 기준**: "본인이 일상적으로 DOL을 켜고 학습을 이어갈 만큼 결함 없음". 정량 지표 없이 **주관 판정** — 결함 발견 빈도가 주당 0~1건 이하로 수렴.

### P1 — 개인 학습 실사용 (전 언어)

**목표**: 5개 카테고리 전체를 본인이 학습 목적으로 **실제 사용**하며, DOL이 자기 학습 기본 도구로 정착되는지 검증.

**활동**:
- 카테고리별 최소 X개 챕터 학습 (목표치는 P0 완료 시점에 정함)
- `/debug/telemetry` 이벤트 + 주관 메모를 축적
- 개선 요청·결함 발견 시 별도 backlog에 적재 후 P0에 재유입

**P1 → P2 전환 기준** (선택):
- 일정 기간(예: 4주) 이상 사용 지속 여부.
- "마크다운/eBook 대비 체감 학습 효과" 본인 판정 — 효과 있음 또는 중립이면 P2 진입, 효과 없음이면 **P2 전환 보류 + 도메인 용도 재검토**.
- 효과 판정은 1주 슬라이스 대조군이 아니라 **일상 사용 중 누적 감각**으로. 이 부분이 기존 설계와 가장 크게 다른 지점.

### P2 — 회사/팀 사용 확장

**개시 조건**: P1 성과가 DOL이 팀 자산으로 가치 있다고 본인이 판단.
**설계 산출물**: 별도 문서 `docs/YYYY-MM-DD-team-version-design.md` 착수.
**고려 영역 (이 문서 시점 미확정)**:
- 멀티 유저 / 권한 분리 / SSO / 조직 콘텐츠 네임스페이스
- 팀 내부 콘텐츠 저작 플로우 (docs-source 대체 or 확장)
- 팀 계측 집계 (팀 대시보드)
- 자가 호스팅 옵션 (GitHub Pages 대신 팀 서버)

### P3 — 공개 버전

**개시 조건**: P2 안정 + 팀 외부 수요 신호 확인.
**설계 산출물**: 별도 문서 `docs/YYYY-MM-DD-public-version-design.md`.
**고려 영역**:
- 인증 / 결제 / 운영 비용 / SLA
- 콘텐츠 기여 / 모더레이션
- 공개 지속가능성 (오픈소스 vs 제품)

## 3. 즉시 영향 (P0 진행 중 현시점에서 해야 할 일)

1. **"1주일 대조군 실험" 게이트 해제**: 작업 일정에서 해당 검증 단계 제거. 설계 문서 `2026-04-18-dev-quest-library-phase2-design.md` §10 참조 시 본 문서가 우선.
2. **P0 잔여 목록 적재**: 위 §2 P0 체크리스트를 backlog로 유지. 현재 세션에서는 별도 추적 도구 없이 문서 내 `- [ ]` 항목으로 관리.
3. **다음 세션 재개 지점**: P0 잔여 중 우선순위 높은 항목부터 PR 분할해 순차 진행. 후보:
   - Phase 3 파서 잔여 19건 "other" 패턴 분석 → 필요 시 PR #8
   - `integration` 잡 strict 승격 PR
   - 타 카테고리 시뮬레이터 타입 설계 (설계 문서 先, 구현 後)

## 4. 판단 근거 (사용자 지시 인용)

> "개인 사용용으로 모든 기능이 전부 완료가 된 다음 모든 언어별로 개인 학습용으로 모두 사용후
>  회사 또는 팀으로 사용 하는 구성으로 확장과 불특정 다수가 사용하는 공용 버전을 순차적으로 별도로 계획하기로 변경"

요지: **완성 → 실사용 → 확장**의 직선 경로. 초기 슬라이스 검증 게이트는 이 경로에서 불필요한 병목으로 재평가됨.
